# ESLint Baseline

This file records **known pre-existing lint violations** as of the baseline snapshot. CI only fails on **new** violations not in this file.

## Why?

When lint rules are added or upgraded, pre-existing violations in existing code block all PRs until fixed everywhere. The baseline mechanism allows gradual remediation without blocking unrelated work.

## Usage

### Check for new violations (CI)
```bash
npm run lint:check
```
Exits 0 if no new violations, exits 1 if new violations found.

### Update baseline after fixing violations
```bash
npm run lint:baseline
```
Regenerates `.eslint-baseline.json` with current violations. Commit the updated file.

## Workflow

1. **Normal development**: Run `npm run lint` locally. Fix violations in code you touch.
2. **CI gate**: Automated check runs `npm run lint:check`. Only NEW violations fail the build.
3. **After fixing violations**: Run `npm run lint:baseline` to update the baseline, then commit.

## DO NOT

- ❌ Update baseline to "hide" new violations you introduced — fix them instead
- ❌ Commit baseline changes unrelated to actual violation fixes
- ✅ Update baseline ONLY after you've actually fixed violations

## Files

- `.eslint-baseline.json` — Baseline violations (committed)
- `.eslint-current.json` — Temp file during check (gitignored)
- `scripts/lint-baseline-check.mjs` — Baseline comparison script
